# Agent context template

A lightweight starting point for new projects that use AI agents. This repo has no app code—just a small set of docs that tell agents how to find context (what you’re working on, how the project is structured, where things run) so they can pick up quickly in new chats.

Copy or clone this repo, run the one-time setup, then add your own modules and work-in-progress docs as you go.

---

## What’s in this repo

**At the project root**

- **README.md** (this file) — Overview and how to use the template. For you.

**In `agent/`**

Shared docs that define the workflow and stay the same across projects:

| File | Purpose |
|------|---------|
| **current-state.md** | For agents. Project-level entry: what the project is, how it's organized, where to read next. |
| **intro.md** | Entry point for agents: where to get WIP state, module state, run environment; read order and rules. You point agents here plus a WIP doc path. |
| **new.md** | How to start a new work-in-progress (WIP) or create a new module (folder + README + context). |
| **cleanup.md** | End-of-session: sync READMEs and WIP docs, push to the WIP branch. Use when you say “clean up according to cleanup.md”. |
| **setup.md** | One-time init after copying this template: regenerate `system-environment.md` for your machine, optionally update the index. |
| **index.md** | Index of all docs with short descriptions so agents (and you) can find the right file without scanning the tree. |
| **system-environment.md** | Run environment for this project (OS, interfaces, config). Regenerated by setup when you move to a new machine. |

**Modules**

The template includes one **example module** (`example-module/`) so you can see the layout: README, **context/current-state.md**, **context/wip-example.md**, and a trivial code file. It also contains **example-module/example-submodule/** to show that **nesting is part of the format**—same structure (README, context/, code) at every level. When you run setup for a new project, the agent **deletes** the whole **example-module/** tree so your repo starts with no modules. You then add real modules using **agent/new.md**.

Each module is a folder (at repo root or nested inside another module, e.g. `my-service/`, `parent/sub-module/`) with:

- A **README** for you (overview, how to run it).
- A **context/** folder with a **current-state.md** for agents (what the module does, rules, what’s working) and WIP docs (e.g. `wip-add-feature.md`).

---

## How to use this template

1. **Copy or clone** this repo into your new project folder.
2. **One-time setup** — In a chat, point the agent at **agent/setup.md** and ask it to set itself up. It will regenerate **agent/system-environment.md** for your environment and update the index if needed.
3. **Add work** — When you’re ready to build something, use **agent/new.md**: the agent can create a new module (new folder + README + context) or a new WIP doc in an existing module.
4. **Daily use** — For new chats, give the agent **agent/intro.md** and the path to the WIP doc you’re working on (e.g. `my-service/context/wip-add-api.md`). It will follow the read order and load the right context.
5. **End of session** — When you’re done, ask the agent to “clean up according to cleanup.md” so it syncs docs and pushes to the WIP branch.

More detail lives in **agent/intro.md** and **agent/setup.md**; the agent uses **agent/index.md** to find the right doc without opening everything.
